# 为什么我把 Claude 拆成 5 个角色

> 一份给程序员的 AI 协同工作流升级指南。
> 不需要照搬框架，理念和方法本身就值得借鉴。

---

## 你是不是经常遇到这些场景

- 跟 Claude 说一句需求 → 它直接给方案 → 实现到一半发现需求理解偏了
- 写到一半改主意 → 重新解释一遍上下文 → Token 耗在「重述」上而不是「干活」
- 一次让它做完整功能 → diff 一上来 200 行 → Review 时只能粗看
- 同一个对话越聊越乱 → 不知道哪一句是当前要做的事
- 跨会话继续上次的工作 → 必须把项目背景重讲一遍

**这些不是 Claude 不够强，是你给它的工作方式有问题。**

---

## 核心洞察

> **让 AI 在明确的角色边界内工作，比自由对话效率高 3-5 倍。**

这不是 Claude 的特性，而是**认知科学原理**。

人类大脑解决复杂问题时，会自然地切分阶段：
- 先想清楚要什么（**澄清**）
- 再分析根因（**诊断**）
- 然后设计方案（**规划**）
- 接着执行（**实现**）
- 最后回看（**复盘**）

而我们跟 Claude 对话时，往往让它**同时做这五件事**——结果就是每件都做得不够好。

**解决方案**：把这五个阶段拆成五个独立角色，每次对话只让 Claude 扮演一个。

---

## 五个角色

```
用户       /pm  →  /diagnose  →  /design  →  /execute  →  /wrap  →  用户
          澄清    诊断          规划       实现         复盘
```

| 角色 | 触发词 | 一句话职责 |
|---|---|---|
| **PM**       | `/pm`       | 把你混乱的需求整理成结构化清单 |
| **Diagnose** | `/diagnose` | Bug 根因诊断，只定位不修复 |
| **Design**   | `/design`   | 产出任务卡（只写 What，不写 How） |
| **Execute**  | `/execute`  | 按任务卡实现代码，独立 worktree |
| **Wrap**     | `/wrap`     | Review diff + merge + 归档 + 写记忆 |

**每个角色都有明确的「不做什么」清单**。
比如 PM 不诊断、Diagnose 不出方案、Design 不写实现细节——**越界即退出角色**。

---

## 对比：自由对话 vs 角色驱动

| 维度 | 自由对话 | 角色驱动 |
|---|---|---|
| **思维边界** | 同时思考"是什么/怎么做/对不对" | 一次只做一件事 |
| **Token 效率** | 大量 token 消耗在"重述上下文" | 通过文档交接，零重述 |
| **产物追溯** | 散在对话里，难审计 | 每个阶段固定输出物（Brief / 任务卡 / Review 报告） |
| **跑偏风险** | 实现到一半发现理解错了 | 每个阶段有强制中断点，错了能早叫停 |
| **并行能力** | 一次只能聊一件事 | 多个任务可并行（多 worktree） |
| **跨会话** | 必须重讲背景 | 记忆系统 + 文档持久化 |

## 五个值得借鉴的设计原则

**即使你不用我这个框架，这些原则也能直接用在你的日常工作里：**

### 1. 认知分工，不是执行分工

> 让 Claude 思考"要什么"和"怎么做"，应该是**两次不同的对话**。

写一个任务规格再让它实现，比"边想边写"质量高得多。

### 2. 任务卡只写 What，不写 How

> 任务卡里如果出现了具体伪代码、SQL 语句、变量名——你已经把实现的活干了。

正确的任务卡：函数签名 + 行为约束 + 验收清单。
让 Claude 自己读代码决定怎么实现，**它会做得比你想象的更符合项目风格**。

### 3. 副作用延迟（适用所有 AI 建议类操作）

> AI 给的建议先不要立即写库 / 改文件 / 触发外部调用。
> 先存到内存或临时状态，**等用户显式确认后再生效**。

避免"用户还没看 AI 就已经改了数据库"。

### 4. Prompt 防御纵深

> Prompt 里的约束（条数限制、格式要求）必须在**代码层同步兜底**。

不要相信 LLM 100% 遵守 prompt。"最多 3 个" 就在代码里 `[:3]`，"返回 JSON" 就在代码里 `try/except`。
原则：**Prompt 违规时静默修正，不崩溃**。

### 5. 文档交接，不口头交接

> 阶段之间用结构化文档传递信息，而不是"接着上次说的"。

PM 输出 Brief 文件 → Design 读 Brief → 任务卡文件 → Execute 读任务卡。
**每个文档都是可恢复的检查点**，跨会话也能继续。

## 实战效果（来自 MyPresent 项目）

这套工作流在 [MyPresent](https://github.com/wjq19990427/MyPresent) 项目中经过 **60+ 个完整任务卡**验证：

- **任务平均粒度**：5-50 行 diff（每张卡职责清晰）
- **并行执行**：常规 2-3 个 worktree 同时推进
- **Review 通过率**：第一次提交即通过 ≈ 85%（剩余 15% 在 Wrap 阶段精准定位问题，10 行内修复）
- **跨会话恢复成本**：≈ 10 秒（读 STATUS.md + 当前任务卡）

## 即使不用我的框架，你今天就能开始做的事

不需要安装任何东西，也不用改你的项目结构：

1. **下次让 Claude 做需求时，先让它整理需求清单，不要让它直接给方案**
   - 即使没有 `/pm` skill，你也可以说："不要给方案，先帮我把需求拆清楚"
2. **写代码前先写一份任务卡**（哪怕只是对话里的一段文字）
   - 必须包含：改动范围 + 接口约定 + 不要做的事
3. **永远在新分支或 worktree 工作，不要直接改 main**
   - 这是基操但常被忽略
4. **每次 merge 前对照任务卡逐项 review，不要凭感觉**
   - 哪怕只是过一遍清单
5. **重要决策和用户偏好沉淀到一个文件里**，下次对话开始时让 Claude 读
   - 即使没有完整的记忆系统

## 后续可拓展的方向（企业场景化）

这个框架是骨架，**所有角色都可以根据公司需求叠加约束**：

### 1. Wrap 角色叠加安全审计

```
默认 Wrap 工作流：Review → merge → 归档 → 记忆
企业增强版：     Review → 安全扫描 → 合规检查 → merge → 归档
```

在 `roles/wrap.md` 中追加：
- 必须执行 SAST 工具扫描
- 检查是否引入新依赖（需走采购流程）
- 检查是否写入敏感字段日志

### 2. 增加新角色

适合你团队的场景：
- `/security` — 安全审查角色，专门看 OWASP Top 10
- `/perf` — 性能审查角色，关注复杂度和资源消耗
- `/migrate` — 数据迁移专用角色，强制要求 dry-run 验证
- `/spike` — 探索性原型角色，允许"丑代码"但限定 24h 内归档

### 3. 团队场景：不同人扮演不同角色

- 资深工程师扮演 PM + Design + Wrap
- 初级工程师扮演 Execute（在严格任务卡约束下）
- 安全工程师专门触发 `/security`

每个人都用同样的 AI 工具，但**角色约束保证产出质量**。

### 4. CI/CD 集成

- Wrap 通过后自动触发部署 pipeline
- 任务卡归档触发 Linear / Jira 状态同步
- 记忆系统接 Slack 知识库

### 5. 多 AI 协同

虽然这个框架默认单 Claude 多角色，但完全可以：
- Claude 扮演 PM / Diagnose / Design / Wrap（思考密集型）
- Codex / Cursor 扮演 Execute（实现密集型）
- 不同 AI 读同一套 `roles/{role}.md` 文档

如果你今天只能记住一句话：

## Q&A 预想

**Q: 这跟直接写需求文档再让 Claude 实现有什么区别？**
A: 区别在「强制中断点」和「文档交接」。直接写需求文档容易陷入"我已经想清楚了"的错觉，PM 角色会反问你没想清楚的部分。

**Q: 切角色不会很麻烦吗？**
A: 你只需要在对话里输入 `/pm` `/design` 这种触发词。比手动复制需求文档快得多。

**Q: 一个小 Bug 也要走完整 5 个角色？**
A: 不需要。小修复可以跳过 PM/Diagnose，直接 `/execute` + `/wrap`。框架支持灵活裁剪。

**Q: 如果团队里只有我用 Claude，怎么推广？**
A: 不需要团队都用框架。**这些原则你一个人也能用**——任务卡可以是你 git commit 前给自己的清单。
