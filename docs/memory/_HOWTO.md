# 会话内存系统说明

> 长期项目的核心组件。跨会话保持用户偏好、项目语境、协作历史。

## 四类记忆文件

| 文件名 | 内容 | 更新时机 | 例子 |
|---|---|---|---|
| `user.md` | 用户工作风格、偏好、背景、技术水平 | 有新认知时 | "独立开发者，偏好结论先行" |
| `feedback.md` | 协作过程中的偏好、习惯、边界 | 用户明确纠正或确认时 | "任务卡只写 What，不在对话展开实现" |
| `project.md` | STATUS.md 放不下的长尾背景、决策依据 | 有重要语境变化时 | "2核2G服务器，必须避免重 Embedding" |
| `reference.md` | 外部资源指针：文档链接、工具、API | 按需 | "Linear 项目 / Grafana 监控 URL" |

## 与 `docs/STATUS.md` 的区别

| 维度 | STATUS.md | memory/ |
|---|---|---|
| 关于 | 代码和任务 | 人和协作方式 |
| 内容 | 当前版本、架构、最近任务 | 用户偏好、项目长尾背景 |
| 更新者 | `/wrap` 每次任务后 | `/wrap` 有沉淀价值时 |
| 上限 | ≤ 50 行 | 每文件 ≤ 200 行 |

## 写入规则（仅 `/wrap` 角色）

### 决策树

```
本次任务有什么洞察吗？
  ↓
对未来还有用吗？（不是单次任务的临时上下文）
  ↓ 是
能从代码 / git log / grep 实时查到吗？
  ├─ 能 → 不写（让未来 AI 实时查更准）
  └─ 不能 → 值得记
       ↓
       属于哪类？→ 写入对应文件
```

### 写入格式

```markdown
## {规则标题}
{规则本身，简洁一句}

**Why：** {用户给出的理由或发生过的事故}
**How to apply：** {何时何地适用}
```

**Why + How to apply 是必填**——让未来 `/wrap` 能判断这条记忆是否还适用。没有 Why 的规则就是教条，迟早过时。

---

## 索引文件约束

- `MEMORY.md` 每行 ≤ 150 字
- 每类只有一个文件，不拆分（不允许 `user_v2.md`）
- 内容变更直接覆写对应行，**不追加历史版本**（历史在 git 里）
- 项目启动时 MEMORY.md 可以为空，随工作流逐步沉淀

---

## 各角色读写权限

| 角色 | 读 | 写 |
|---|---|---|
| `/pm` | ✅ user.md、feedback.md | ❌ |
| `/diagnose` | ✅ project.md、reference.md | ❌ |
| `/design` | ✅ user.md、feedback.md、project.md | ❌ |
| `/execute` | ❌ 不读（避免干扰实现专注） | ❌ |
| `/wrap` | ✅ 全部 | ✅ 唯一写入者 |

**为什么 execute 不读？** 实现阶段需要专注于任务卡和代码，记忆里的协作偏好对编码无直接价值，反而是干扰。

## 典型记忆示例

### user.md 示例

```markdown
# User Profile

## 角色
独立开发者 + 产品经理

## 工作风格
- 偏好结论先行，列表式输出
- 决策喜欢看 trade-off 而非单一方案
- 实施时希望先讨论再动手，但小修改可直接做

**Why：** 用户在多次对话中明确表达过这些偏好
**How to apply：** 写任何报告/分析时直接结论先行；提方案时给 2-3 个候选 + 推荐
```

### feedback.md 示例

```markdown
# Collaboration Feedback

## 任务卡只写 What
任务卡严禁出现伪代码、SQL、具体 widget 类型。

**Why：** 用户发现写 How 会让执行者要么复制粘贴，要么忽略，浪费 token
**How to apply：** /design 角色自查时严格执行

## Review 必须正式
每次 merge 前必须有结构化 Review 表格 + 明确 PASS/FAIL。

**Why：** 用户希望 merge 决策可追溯
**How to apply：** /wrap 角色不跳过 Review
```

---

## 维护原则

1. **少而精**：每条记忆都有明确的 Why，没 Why 的不写
2. **及时清理**：过时的记忆主动删除，不堆积
3. **索引精简**：MEMORY.md 不会一直涨，更新条目时直接替换
4. **跨项目复用**：用户偏好可以跨项目继承，项目背景不行
