# 如何编写 L2 契约文档

> L2 契约是模块公开 API 的**唯一权威来源**。执行者改代码前必须读这里，改了公开签名/副作用语义后必须同步更新这里。

## 文件命名

每个代码层一个文件：`docs/api/{layer}.md`

例如：`docs/api/core.md` / `docs/api/skills.md` / `docs/api/components.md`

## 文件结构模板

```markdown
# {layer}/ — {层职责描述}

> {一句话说明边界规则，如：只消费下层，不实现业务逻辑}

## 边界规则

- {规则 1}
- {规则 2}

## 模块清单

| 文件 | 职责 | 契约状态 |
|---|---|---|
| `foo.py` | {职责} | ✅ / 🚧 待填 |
| `bar.py` | {职责} | ✅ |

---

## foo.py

> {模块一句话描述}

### `function_name(param1: Type, param2: Type) -> ReturnType`

- **入参**：
  - `param1`：{说明}
  - `param2`：{说明}
- **返回**：{说明}
- **副作用**：{DB 写入、文件操作、session_state 变更等}
- **已知陷阱**：{容易踩的坑}

### `class FooClass`

- **职责**：{一句话}
- **关键方法**：列出公开方法及其行为契约
- **状态**：是否持有内部状态？是否线程安全？
```

## 写什么

- 公开函数/类的签名、入参约束、返回结构
- 非显而易见的副作用（写 DB、触发 hook、改全局状态）
- 调用方需要知道的约束（必须在 X 之前调用、不能在 Y 上下文中用）
- 已知陷阱（踩过坑的地方）

## 不写

- 实现细节（那是 L3 源代码的事）
- 注释掉的代码或临时 TODO
- 明显从函数名就能读出来的内容
- 私有函数 / 私有方法（带下划线前缀的）

## 更新时机

| 改动 | 是否需要更新 L2 |
|---|---|
| 新增公开函数/类 | ✅ 必须 |
| 修改函数签名（参数/返回类型变化） | ✅ 必须 |
| 改变副作用语义（如原来不写 DB 现在写了） | ✅ 必须 |
| 改变调用约束（如必须在 X 后调用） | ✅ 必须 |
| 删除公开函数 | ✅ 必须（同时检查所有调用方） |
| 纯内部重构，签名和语义不变 | ❌ 不需要 |
| 修复 bug，不改对外行为 | ❌ 不需要 |
| 改注释/docstring | ❌ 不需要 |

## LLM Skill 类模块的特殊约定

对包含 AI 调用的 Skill / Service 模块，在 L2 契约中必须额外说明：

### 副作用字段要精确区分

```markdown
- **副作用**：
  - 写 `session_state["xxx"]`（Skill 内部，同步）
  - **不**写 DB（DB 持久化由调用方在用户保存时触发）
```

### 单一职责声明

注明本 Skill 只负责哪一类 AI 任务，不跨越边界：

```markdown
> 职责边界：只做内容分析（标题/摘要/感受），不做分类标注（标签/维度）。
```

### Prompt 兜底说明

说明代码层如何兜底 LLM 可能的格式违规：

```markdown
- **Prompt 兜底**：返回列表截断至 `[:3]`；非 list 类型降级为空列表；JSON 解析失败返回失败结果（不抛异常给用户）
```

这些信息是执行者需要知道但读代码未必能立即看清的约束，属于 L2 契约的合法内容。

---

## 维护原则

1. **写入即权威**：L2 一旦写入，代码必须与之一致；不一致 = 代码有 bug 或 L2 过时
2. **拒绝细节蔓延**：L2 是契约不是教程；超过 5 行 docstring 级别的说明应该挪到代码注释或单独文档
3. **审查时核对**：`/review` 角色每次都要检查 L2 是否需要同步
4. **新执行者读 L2 而非读代码**：如果 L2 写得清楚，执行者读完 L2 就能开干；如果还得读源码才知道契约，说明 L2 不合格
